# ADR-0001: Push-based INSERT INTO for Pinot

## Status
Implemented

## Decision summary
- Require explicit target type for hybrid tables.
- Use a unified SQL/API surface with separate row and file execution backends.
- Default row backend is shard log + local prepared store on the server.
- Default file backend is Minion/segment generation + statement manifest + lineage-backed publish.
- Add SPI interfaces in `pinot-spi` under `org.apache.pinot.spi.ingest` and default implementations in `pinot-core` and `pinot-controller`.
- Persist row prepared data under server `dataDir/insert/prepared/`, not consuming-segment files.
- Expose full lifecycle states: NEW, ACCEPTED, PREPARED, COMMITTED, VISIBLE, ABORTED, GC.
- Three consistency modes: FIRE_AND_FORGET, WAIT_FOR_ACCEPT (default), WAIT_FOR_VISIBLE.
- Keep the implementation on by default, but validate unsafe cases (partial upsert, dedup) instead of silently accepting them.

## Open questions (resolved)

### Exact admin/status API shape
**Resolution:** Four REST endpoints on the controller:
- `POST /insert/execute` -- submit an INSERT request
- `GET /insert/status/{statementId}?table={tableNameWithType}` -- check status
- `POST /insert/abort/{statementId}?table={tableNameWithType}` -- abort a statement
- `GET /insert/list?table={tableNameWithType}` -- list all statements for a table

Additionally, `POST /insert/rows` on the server for internal row dispatch.

### Final package names for SPI and default implementations
**Resolution:**
- SPI interfaces: `org.apache.pinot.spi.ingest` (InsertExecutor, InsertExecutorFactory, InsertRequest, InsertResult, InsertStatementState, InsertConsistencyMode, InsertType, ShardLog, ShardLogProvider, PreparedStore, ArtifactPublisher, PublishCoordinator)
- Row executor and recovery: `org.apache.pinot.core.data.manager.ingest` (RowInsertExecutor, RowInsertExecutorFactory, InsertRecoveryManager, InsertRowApplier, InsertPartitionRouter, LocalShardLog, LocalPreparedStore, GenericRowSerializer)
- File executor and coordinator: `org.apache.pinot.controller.helix.core.ingest` (FileInsertExecutor, FileInsertExecutorFactory, InsertStatementCoordinator, InsertStatementManifest, InsertStatementStore)
- SQL parsing: `org.apache.pinot.sql.parsers.parser.SqlInsertIntoValues` and `org.apache.pinot.sql.parsers.dml.InsertIntoValues`

### Whether v1 should expose synchronous WAIT FOR VISIBLE
**Resolution:** Yes. `InsertConsistencyMode.WAIT_FOR_VISIBLE` is defined and available. The default is `WAIT_FOR_ACCEPT`. Clients that need read-your-writes semantics can use `WAIT_FOR_VISIBLE` at the cost of higher latency.

### How much OFFLINE row micro-batching logic should be configurable
**Resolution:** In v1, row batching is implicit per-statement (all rows in a single INSERT statement are batched together). Cross-statement micro-batching is not implemented. The partition router uses the table's configured partition function to route rows.

## Implementation notes

### Statement lifecycle
The full state machine is: NEW -> ACCEPTED -> PREPARED -> COMMITTED -> VISIBLE -> GC, with ABORTED reachable from ACCEPTED, PREPARED, or COMMITTED. The coordinator manages all transitions and persists manifests in ZooKeeper.

### Idempotency
Request-level idempotency is provided via the `requestId` + `payloadHash` mechanism. Same requestId + same payload returns the existing result. Same requestId + different payload returns IDEMPOTENCY_CONFLICT.

### Table mode safety
File inserts reject partial-upsert and dedup tables. Full-upsert tables require partition configuration. Row inserts require valid partition function configuration for upsert/dedup tables.

### Recovery
Server-side InsertRecoveryManager scans the prepared store on startup, queries the coordinator for authoritative state, and takes action (replay, cleanup, or leave for timeout). Exponential backoff retry (1s-30s, 5 attempts) handles transient coordinator unreachability.

### Background cleanup
Coordinator runs a cleanup sweep every 5 minutes:
- ACCEPTED/PREPARED stuck > 30 min: aborted
- COMMITTED stuck > 60 min: aborted
- VISIBLE/ABORTED > 24 hours: GC (manifest deleted from ZK)

### Test coverage
171 unit tests across pinot-common (SQL parsing), pinot-core (row executor, shard log, prepared store, partition routing), and pinot-controller (coordinator, file executor). No integration tests in v1.
